## Section 3 — Alternative Means of Recording Work Time Compliance Requirements

### Requirement 1: Data Collected

> *The data to be collected must be sufficient to enable adequate monitoring of work time.*

Drive Legal records the following data elements for every shift event, satisfying all data requirements specified in the Rule:

| Required Data Element | Drive Legal Implementation |
|---|---|
| Driver's name | Captured at registration; displayed on all records and enforcement output |
| Date on which the logbook page starts | Recorded as ISO 8601 timestamp at shift start; displayed as "Today, DD MMM YYYY" |
| Registration plate details | Driver enters vehicle registration at registration; recorded per shift |
| Start and finish distance recorder (odometer) readings for RUC-liable vehicles | Odometer input fields are presented at shift start and shift end; stored per shift log entry |
| Start time with location for each work time entry | GPS coordinates and reverse-geocoded suburb/city name captured automatically at shift start |
| Finish time with location for each work time entry | GPS coordinates and reverse-geocoded suburb/city name captured automatically at shift end |
| Start time with location for each rest break | GPS coordinates and suburb/city name captured at break start |
| Finish time with location for each rest break | GPS coordinates and suburb/city name captured at break end |
| Extent of hours worked in a cumulative work day | Calculated in real time; displayed on dashboard and in enforcement view |
| Extent of hours worked in a cumulative work period | Calculated across the rolling 14-day window; displayed on dashboard (Fortnightly stat card) |
| Days on which no work occurred that are subject to work time | Captured implicitly through the absence of shift records; visible in the Logbook history view |

All data is stored with a UTC timestamp and displayed in New Zealand Standard Time / New Zealand Daylight Time as appropriate (section 3.1.7 compliant).

---

### Requirement 2: Data Transfer, Stability, Exclusivity, and Integrity

> *The data collected must be able to be transferred between components of a system that protects its exclusivity and integrity. Once the data is locked into the system by the driver it must not be capable of alteration in any way.*

Drive Legal implements a **SHA-256 cryptographic hash chain** to ensure tamper-evident record integrity. Each completed shift record is hashed using a canonical JSON representation of the full record, and each hash incorporates the hash of the previous record (chained). Any alteration to any historical record — including timestamps, locations, or odometer values — would break the hash chain and be immediately detectable.

The hash chain is stored both locally on the device (AsyncStorage) and in the cloud database (MySQL/TiDB via HTTPS). The cloud database stores the exact canonical JSON representation used to generate each hash, enabling independent verification.

Once a shift is ended and its record is finalised, the record is locked and cannot be edited. The app does not provide any interface to modify completed shift records. The hash chain verification status is displayed in the Enforcement View and in all PDF exports.

Data is transferred to the cloud server over HTTPS (TLS 1.2+). The server enforces authenticated API access; all sync endpoints require a valid session token. Cloud database access is restricted to the application server and is not publicly accessible.

### Requirement 3: Driver Identification

> *The system must be capable of identifying the drivers who are using it.*

Each Drive Legal account is uniquely identified by:

- **Email address** (unique per account, used as login credential)
- **Full name** (entered at registration)
- **Driver licence number** (entered at registration; displayed on all enforcement output)
- **Vehicle registration** (entered at registration; recorded per shift)
- **Driver type** (Goods Service or Passenger Service; selected at registration)

The system requires a username (email) and password to log in. Each logbook record is permanently associated with the registered driver's name and licence number. The enforcement view and all PDF exports display the driver's full name and licence number on every page.

The system is designed for sole-driver use: each account belongs to one driver, and the logbook is for the sole use of the registered driver (section 4.1.2 compliant).

### Requirement 4: Exclusivity, Uniqueness, and Integrity — Tamper Prevention

> *The application must explain what measures the system will include to prevent any tampering with the system devices, including the driver device if one is used, and the data records held in the system.*

Drive Legal implements the following tamper-prevention measures:

**Cryptographic hash chain:** As described under Requirement 2, every completed shift record is SHA-256 hashed and chained. Any modification to any record — including by a technically sophisticated user with direct database access — would produce a hash mismatch detectable by the verification algorithm. The enforcement view displays the hash chain verification status ("✓ Verified" or "⚠ Chain broken") for every record.

**Locked records:** Once a shift is ended, the record is finalised and the app provides no interface to edit or delete it. There is no "edit history" or "delete entry" function in the app.

**Immutable timestamps:** All timestamps are recorded as UTC Unix milliseconds at the moment of the event. The app does not allow the driver to manually enter or override timestamps for current events (only for retrospective entries, which are flagged as such in the record).

**Server-side storage:** Completed shift records are pushed to the cloud database over HTTPS. The server does not expose any endpoint to modify or delete completed records. Records in the cloud database are append-only from the application's perspective.

**Account authentication:** The app requires email and password authentication. Passwords are hashed using SHA-256 before storage. Session tokens expire and are not stored in plaintext.

**No overwrite of auto-populated data:** GPS coordinates, timestamps, and hash values are populated automatically by the system and cannot be overwritten by the driver (section 3.1.4 compliant).

---

### Requirement 5: Enforcement Officer Access

> *The system must be accessible to enforcement officers so records can be viewed roadside or in depot and retained for checking and possible investigation and prosecution, in a timely and cost-effective way.*

Drive Legal provides enforcement access through two mechanisms:

**In-app Enforcement View:** A dedicated "Enforcement View" screen (accessible from the Logbook tab) displays the full cumulative work period in a structured grid format, including: end of last 24-hour break, work time hours today, total hours in the cumulative work period, all shift events with timestamps and locations, odometer readings, rest breaks, and hash chain verification status. This view is designed to be readable on a mobile screen at the roadside.

**PDF Export:** The driver can generate a PDF report from the Reports tab at any time, including at the roadside. The PDF includes: driver name and licence number, vehicle registration and odometer readings, all shift events with start/end times and GPS locations (suburb/city), cumulative hours (daily and fortnightly), rest breaks, hash chain integrity status, and the Drive Legal version and generation timestamp. The PDF can be shared by email, AirDrop, or any other sharing method available on the device.

**Email sharing:** The PDF export uses the device's native share sheet, allowing the driver to email the PDF directly to an enforcement officer or Waka Kotahi at the roadside.

**Waka Kotahi testing access:** Permanent testing access is provided via the credentials listed in Section 2 above. This access remains available at no cost to Waka Kotahi indefinitely.
