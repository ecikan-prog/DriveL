# NZTA Review Notes

## Source 1: Waka Kotahi Electronic Logbook Minimum Specifications (Sept 2022)

### Pages 1-5 key findings

#### Approval principles
- Section 2.1: The minimum information required by law must be recorded.
- Section 2.2: Copies of one or more logbook records must be immediately available/visible or extractable and able to be provided by email at roadside to an enforcement officer or Waka Kotahi.
- Section 2.5: Chain of evidence requirements must be preserved.
- Section 2.6: Each record must be uniquely identifiable.
- Section 2.7: The system must remind drivers when rest breaks or rest periods are due.
- Section 2.9: Approval uses a 3-stage assessment: self-check by applicant/supplier, Waka Kotahi assessment, and NZ Police assessment.

#### System / performance / audit / backup
- Section 3.1.1: System must include an automatic self-check procedure to ensure it is working as it should.
- Section 3.1.2: System must be synchronised in real time between all delivery platforms and servers when multiple devices are used in one cumulative work day/period.
- Section 3.1.3: Must be able to receive and apply software updates required by future legislative or technology changes.
- Section 3.1.4: Must not allow overwriting of data entered automatically by the system.
- Section 3.1.5: Must include a change/audit log that automatically records date/time of any changes to the system, including routine software changes.
- Section 3.1.6: Must provide a non-erasable record of changes to built-in settings such as clocks and login codes.
- Section 3.1.7: Must accommodate NZST and NZDT changes.
- Section 3.1.8: Must include the actual date/time stamp (not alterable) for each entry.
- Section 3.1.9: Must show the driver name and driver licence number.
- Section 3.1.10: Must reflect a unique identifier on each cumulative work day entry.
- Section 3.1.11: Suitable external and secure backup of all data must be available.

#### Data and work time entries
- Section 3.1.12: Employer/operator can set up, install, collect, and maintain the driver’s information.
- Section 3.1.13: Self-employed drivers can set up, install, collect, and maintain their own information.
- Section 3.1.14: Security requires logbook login using username and PIN/password.
- Section 3.1.15: System requires deliberate user action to commence and end cumulative work day.
- Section 3.1.16: If system remains logged in permanently to a device, a PIN/password prompt is required to confirm the driver midway through the CWD.
- Section 3.1.17: Where GPS or system auto-populates fields such as address or date/time, the system must allow correction/amendment before completion/finalisation.
- Section 3.1.18: Must allow notes where there is loss of reception/no signal.
- Section 3.1.19: Must prevent entries for one cumulative work day when a prior work day record is still open.
- Section 3.1.20: Prevents change to cumulative work day records when closed or when a new entry has been made after a work/rest status change.
- Section 3.1.21: If record date/time is amended by a driver, the system must record both the original and changed time and highlight the change in enforcement/summary view.
- Section 3.1.22: Where a record overlaps midnight, the system must record the cumulative work day in one full logbook entry, not as calendar days.
- Section 3.1.23: Must allow retrospective entries for previous days under defined scenarios.

#### Logbook and work time requirements
- Section 3.2.1: Must differentiate driver duty status between work time and rest time.
- Section 3.2.2: Must indicate whether work time record commenced midnight or noon unless otherwise apparent.
- Section 3.2.3: Records date, start/finish time, and location of each work time entry.
- Section 3.2.4: Records date, start/finish time, and location of all rest breaks.
- Section 3.2.5: For all vehicles subject to work time, records the registration plate of each vehicle driven.
- Section 3.2.6: For RUC-liable vehicles, prompts for registration plate and start/finish distance recorder readings for all vehicles driven in a cumulative work day.
- Section 3.2.7: Must provide a place for notes against each entry and preserve those notes as part of the permanent record.
- Section 3.2.8: Must accommodate changes to work status and vehicles driven during a working day.
- Section 3.2.9: Must accommodate multiple Transport Service Licence types: Goods, large passenger, small passenger, and vehicle recovery services.
- Section 3.2.10: Must accommodate vehicle and non-vehicle related work activities, with non-driving work placed in the same way as driving and notes permitted to indicate 'other work, non-driving'.
- Section 3.2.11: Allows the last 24-hour break to be added on first use.
- Section 3.2.12: Must warn drivers if legal work time maximums have been exceeded but does not preclude data entry; includes warnings for mandatory 10-hour rest break and 24-hour rest break.
- Section 3.2.13: Must reconstruct daily records back to beginning of cumulative work period, including most recent cumulative work day.
- Section 3.2.14: Can record any requirements specified in the Land Transport Rule: Work Time and Logbooks 2007, including notes on exemptions/failures.

#### Employer requirements
- Section 3.3.1: At least once in every 24-hour period.
- Section 3.3.2: Electronic logbook records must be available to transport operator in a form identical to that available to enforcement, in unaltered form.
- Section 3.3.3: Electronic logbook records and any additional data received by a transport operator must be stored for at least 12 months from the date of each entry.
- Section 3.3.4: Data must include the date and time it was provided to the transport operator.
- Section 3.3.5: Must notify transport operator when the system/device has not updated within 24 hours, except in the stated break scenario.
- Section 3.3.6: Must accommodate notification of driver work time breaches in a timely manner.

#### Warnings/prompts
- Section 3.4.1: Warning/alert prior to rest break at 5.5/7 hours worktime accrued (5 and/or 15 minutes prior to event).
- Section 3.4.2: Warning/alert prior to end of cumulative work day at 13 hours (5 and/or 15 minutes prior).
- Section 3.4.3: Warning/alert prior to end of cumulative work period at 70 hours (5 and/or 15 minutes prior).
- Section 3.4.4: Warning/alert recommending work time where 30 minute rest break has not been completed.
- Section 3.4.5: Warning/alert recommending work time where 10 hour rest break has not been completed.
- Section 3.4.6: Warning/alert recommending work time where 24 hour rest break has not been completed.

#### Driver summary / display
- Section 4.1.1: Must show approval by Waka Kotahi and date of approval.
- Section 4.1.2: Must display notice that logbook is for sole use of the registered driver.
- Section 4.1.3: Must note when the last 24-hour break ended.
- Section 4.1.4: Must show commencement date/time of active cumulative work period.
- Section 4.1.5: Must show commencement time of active cumulative work day.
- Section 4.1.6: Must show duration of active cumulative work day.
- Section 4.1.7: Must show duration of active cumulative work period.
- Section 4.1.8: Must show next break due and end of cumulative work day/period as applicable.
- Section 4.1.9: Must warn the driver when work time requirements have been exceeded and warning remains visible until compliance restored.

#### Enforcement / output
- Section 5.1.1: Enforcement view must include end of last 24-hour break, work time hours today, and total hours in cumulative work period.
- Section 5.1.2: Active grid must populate for full cumulative work period, horizontal or vertical layout acceptable depending on system design.
- Section 5.1.3: Must reconstruct daily records back to beginning of cumulative work period, including most recent cumulative work day.
- Section 5.1.4: Must provide electronic copies roadside of the active cumulative work period, including most recent cumulative work day.
- Section 5.1.5: Over-midnight records remain one full logbook entry, not calendar days.
- Section 6.1.1: Minimum output data includes date, driver name, licence number, registration plate, hub/ODO reading depending on vehicle type, location details, signature line, sign out/log off if applicable, cumulative work time duration, notes, and details of any changes made by driver.
- Section 6.1.2: Data must be provided in one Microsoft Excel file up to 20MB; if multiple individuals requested, separate based on driver records if file cannot be sent at once.
- Section 6.1.3: Records must be collated as full cumulative work periods.
- Section 6.1.4: Where build data is sought, information must be securely stored and transferred, such as password-protected documents or equivalent secure process.

## Source 2: Waka Kotahi Electronic Logbook Application Form

### Pages 1-4 key findings

The cover page requests the core application identity fields: date, applicant and company name, contact person, email, phone, system make, system model, and operating platform(s) iOS/Android/both. The document is version 1.2, issued October 2021.

The definitions section identifies the applicant as the person and/or organisation applying to have an alternative means of recording matters relevant to the monitoring of work time approved. It also defines the system broadly to include the software and all related parts required to record, retain, and evidence work-time records.

The general information section states that approval is required before any such system may be supplied to the New Zealand market. It notes that Waka Kotahi may approve any number of alternative systems and that the applicant should consider the commercial implications before investing further in the system.

The legal references section is especially important for the application narrative. It states that the application must include information about five matters: how the data is collected; how data is transferred and subsequently managed by the operator; how drivers using the system will be identified; what measures are in place to prevent tampering with devices or records; and how enforcement officers may access records.

The application form also states that applicants must note the relevant provisions of the Fair Trading Act 1986 and Consumer Guarantees Act 1993 applying to the supply and installation of such systems in New Zealand, and that the applicant is responsible for ensuring the system does not breach copyright, trademark, patent, or other protection.

The operational field trial section explains that an initial approved application may still require a field trial before approval is issued, particularly to ensure roadside compliance and usability. The purposes of the field trial are to test compatibility with Police and Waka Kotahi administrative and IT processes, confirm the system can provide documentary evidence of the user's work time to enforcement officers, and confirm that the documentary standard is robust enough to withstand judicial scrutiny. The field trial, if required, involves Waka Kotahi, Police, at least one transport operator, and at least two drivers.

### Application form pages 5-9 key findings

Page 5 adds important field trial conditions and application instructions. A person cannot be both a driver and the transport operator. The drivers and transport operator involved in any field trial must be independent of the applicant, and they must be acceptable to both Police and Waka Kotahi. Enforcement officers must be able to obtain copies of records from the proposed alternative system from drivers and operators upon demand. The application must be completed in full, using the most recent form version, and incomplete information may delay consideration or require return for completion. If the applicant is not based in New Zealand, a suitable New Zealand-based agent must also be provided.

Page 6 contains the main applicant details form fields to be completed. Section 1 asks for: name, New Zealand-based contact person name if an organisation, applicant email address, web address of company/eLog, company number, New Zealand contact telephone and mobile numbers, physical address, and postal address if different. Section 2 asks for the proposed system make/model, country of origin, app name downloadable from both Android and Apple platforms including TestFlight links if used, the unique identifying marks or signage that will allow roadside identification of the system with images if it is not an app-based product, provisions for after-sales support, and how user instruction will be provided, with copies of user documents to be supplied with the application. It also asks for testing access credentials: a username, password, and transport operator/employer access username and password. The form suggests that, where possible, the username should include "Waka Kotahi" or "NZTA", and if a driver licence number is used it should be a clearly invalid example number.

Page 7 is the most important structured compliance page. It requires a report detailing how the proposed system meets five compliance requirements: (1) the data collected must be sufficient to adequately monitor work time, including driver name, logbook page start date, registration plate details, start and finish odometer readings for RUC-liable vehicles driven in the cumulative work day, start and finish time with a location for each work time start and end, extent of hours worked in a cumulative work day, extent of hours worked in a cumulative work period, and days on which no work occurred that are subject to work time; (2) the collected data must be transferred between system components in a way that protects stability, exclusivity, and integrity, and once locked by the driver it must not be alterable; (3) the system must be able to identify the driver using it; (4) the application must explain measures to prevent tampering with the system devices and data records, including where the driver device is used; and (5) the system must be accessible to enforcement officers so records can be viewed roadside or in depot and retained for checking and possible investigation and prosecution, with timely and cost-effective interchange.

Page 8 explains the complete application submission package. The completed application is to be emailed to DriverandOperator@nzta.govt.nz. The package must include: the completed application form; a completed Waka Kotahi Electronic Logbook Minimum Specifications document, which is sought for signing once the system has been tested and commentary finalised prior to approval; a username and password for Waka Kotahi approver access to test and/or review the system at any time, with access provided at no cost and able to remain indefinite; two cumulative work periods as per email with the application form; a copy of user documentation and any other supporting material relevant to the application; any other related information Waka Kotahi should consider; and the signed declaration.

Page 9 is the declaration page for the application form. It requires the named applicant or organisation to certify that the information is true and correct and that the applicant has authorised the application, read and understood the minimum specifications, read and understood relevant Fair Trading Act and Consumer Guarantees Act provisions, read and understood the relevant Land Transport Rule: Work Time & Logbooks 2007 and Land Transport Act 1998 sections, and confirmed that the approval being sought does not breach copyright, trademark, patent, or other protection.
